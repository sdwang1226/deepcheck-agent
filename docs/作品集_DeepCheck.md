## 5 个关键产品决策

### 1. 架构选型：ReAct → Tool Calling

- **问题：** ReAct 模式下 Agent 稳定性仅 50%，频繁出格式错误
- **决策：** 切换至 Tool Calling 架构，匹配模型原生能力
- **结果：** 稳定性 50% → 100%

### 2. 抗幻觉：规则约束 → 机制保障

- **问题：** Prompt 规则约束来源引用效果差（仅 30%）
- **决策：** 将来源标注下沉到数据返回层强制执行
- **结果：** 引用率 30% → 90%+，零数值编造

### 3. 检索模型选型：对比实验驱动

- **问题：** 基线模型对中文查询命中率仅 50%，不满足上线标准
- **决策：** 评估 3 个备选方案，通过 A/B 实验选定中英双语模型
- **结果：** 命中率 50% → 70%

### 4. 可扩展性设计：配置化扩展

- **问题：** 未来需覆盖更多股票
- **决策：** 设计分索引路由 + 注册表方案
- **结果：** 新增标的只需配置，无需开发

### 5. 评测驱动迭代：量化指标体系

- **问题：** 如何衡量每次迭代的效果
- **决策：** 定义 4 维度评测体系，20 条用例自动化验证
- **结果：** 6 版 Prompt 迭代，每次改动可量化验证

## 迭代方法论

```
v1  基础ReAct        → 能跑但不稳定
v2  Prompt加规则      → 格式改善，但引用差
v3  数据层加来源标注  → 引用率 30%→90%（核心突破）
v4  切换Tool Calling → 稳定性 50%→100%
v5  中文查询翻译      → 部分缓解命中率
v6  Embedding升级     → 命中率 50%→70%（根因解决）
```

**核心教训：能改数据层的不改Prompt，能改架构的不改数据层。**
