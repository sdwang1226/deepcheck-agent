## 二、AI 理解能力（DeepCheck 投研 Agent）

---

### Q1：为什么做这个投研 AI Agent？

| 视角 | 发现 |
|------|------|
| **业务方向** | 长桥已有行情 API、财报 API 及 AI Skill 生态，结构化数据能力强；我想探索在这个生态上补充**非结构化年报语义检索**这一方向的可行性 |
| **用户需求** | 个人投资者核心痛点：数据分散、认知门槛高、信息过载，难以获取有效信息支撑决策 |

**结论：** 做一个用户通过自然语言提问，AI 自动调用行情/财务/年报检索工具，给出 **带来源引用** 的投资分析。

> ⚡ 这既是岗位能力的证明，也是一个真实产品机会的验证。

---

### Q2：怎么做的需求定义？（做什么、不做什么）

**产品形态：** "对话式查询" 而非 "自动报告生成"

> WHY：技术复杂度适中，验证速度快，适应 MVP 阶段验证 RAG + Tool Calling 核心假设。报告生成是锦上添花。

---

### Q3：核心指标是什么？怎么定义的？

| 指标 | 衡量什么 | 评分规则（0/1/2） |
|------|---------|------------------|
| **工具调用准确率** | Agent 能否听懂问题找对工具 | 工具错 / 工具对参数错 / 均正确 |
| **数值准确性** | 有没有编数字 | 没数字 / 有数字没单位 / 有数字有单位 |
| **来源引用完整性** | 能不能追溯出处 | 没提到 / 笼统标注 / 精确标注 |
| **RAG 命中率** | 检索是否命中 | <10字符或纯错误 / 10-50字符 / >50字符且无error |

**评测哲学：**

> 评测不追求完美，追求 **"能驱动迭代决策"**。
> 20 条用例覆盖行情、财报、RAG、混合四类场景，每次改 Prompt 或架构都跑一遍，只要能区分"变好还是变差"就够了。

---

### Q4：为什么选这几个核心指标？

**核心原则：从用户最可能踩的坑倒推，覆盖关键风险。**

| 指标 | WHY |
|------|-----|
| 工具调用准确率 | Agent 第一步是听懂问题 → **底层能力** |
| 数值准确性 | 金融场景对数字零容忍 → **防 LLM 编数字** |
| 回答相关性 | RAG 可能命中错误内容 → **检测质量** |
| 来源引用完整性 | 评测中发现幻觉后 **新增** → 解决信任问题 |

> ⚡ **来源引用是抗幻觉的终极防线：** 即使模型偶尔出错，用户看到来源后能自己验证。

---

### Q5：怎么做的架构选型？

**Agent 架构选型五个核心维度：**

#### ① 推理范式：ReAct → Tool Calling

| 阶段 | 选择 | 结果 |
|------|------|------|
| 初始 | ReAct（社区最主流，可解释性好） | 稳定性 **仅 50%** |
| 分析 | DeepSeek 对文本格式遵循能力不够 | ReAct 成功案例基于 GPT-4 |
| 切换 | **Tool Calling**（DeepSeek 原生支持） | 稳定性 **100%** ✅ |

> ⚡ **核心教训：技术选型不能照搬社区最佳实践，必须结合自己的模型和场景做实测验证。**

#### ② LLM 选型

**方法：先列约束条件 → 排除法筛选**

| 约束条件 | 排除结果 |
|---------|---------|
| 必须支持 Tool Calling | — |
| 中文理解要好 | Gemini ❌ |
| MVP 成本要低 | GPT-4 ❌ |
| Tool Calling 要稳定 | 文心、通义 ❌ |

> ⚡ LLM 选型不是看哪个最强，而是选 **最匹配约束条件** 的模型。选择与场景绑定。

#### ③ RAG 方案（5 维度 × 4 版迭代）

**5 个选型维度：**

| 维度 | 可选方案 |
|------|---------|
| 切分策略 | 机械切分 / 递归切分 / 章节切分 / 语义切分 |
| Embedding | TF-IDF / MiniLM / **BGE-large-zh-v1.5** / text-embedding-3-large / M3E·GTE |
| 向量库 | **FAISS** / ChromaDB / Milvus / Weaviate |
| 检索策略 | 纯语义 / BM25 / 混合检索 / ReRanker / HyDE |
| 索引架构 | 单索引 / **分索引 + 路由** |

**4 版迭代：**

| 版本 | 方案 | 结果 & 发现 |
|------|------|------------|
| v1 | 机械切分 + TF-IDF | ❌ 关键词匹配对中英跨语言完全无效 |
| v2 | 章节切分 → Recursive + TF-IDF | 切分不是瓶颈，**Embedding 才是** |
| v3 | Recursive + MiniLM | 方向正确，英文模型对中文查询命中率 **仅 50%** |
| **v4** | Recursive + **BGE 中英双语** | 解决核心矛盾，命中率 **70%** ✅ |

> 停在 70% 是 **MVP 决策** — 够用就行，后续路径清晰（Reranker + 混合检索）

> ⚡ **核心方法论：每次只改一个变量，用评测数据验证，避免多变量无法归因。**

#### ④ Prompt 策略 vs Fine-tuning

选择 Prompt，而非 Fine-tuning：
- DeepSeek 没有 Fine-tune API
- MVP 阶段需要快速迭代
- 实测 Prompt + 机制保障已达 90%+，不做过度工程

#### ⑤ 可扩展性设计

- **多股票架构：** symbol 分索引 + 注册表路由，新增股票只需配置无需改代码
- **多轮记忆：** ConversationBufferMemory

### Q6：怎么看待幻觉问题？怎么处理的？

**核心观点：** 幻觉不是 bug，是 LLM 的本质特性。核心不是消除幻觉，而是 **把幻觉对用户的影响降到最低**。

**分层防御：**

| 层级 | 策略 | 效果 |
|------|------|------|
| **数据层** | 工具返回值强制带来源标签 | 数据可追溯 |
| **Prompt 层** | 无数据时说"暂无数据"不要猜测 | 减少过度推断 |
| **评测层** | 来源引用率 + 数值准确性指标 | 量化监控 |

> ⚡ **能用机制保障的就不要依赖规则约束。**

**追问：更复杂场景怎么抗幻觉？**
- 事实核查层做交叉验证
- 置信度标注（不确定的显示"建议人工确认"）
- 人机协作（高风险结论须人工审核）

**追问：RAG 检索错了怎么办？**
- 评测层有 RAG 命中率指标
- 用户侧显示引用原文片段，即使 LLM 错了用户也能自行判断

**追问：如何从架构层做硬约束？**
- 后处理校验：回答中数字与工具返回值交叉验证
- 白名单模式：只允许输出工具返回值中的数据
- 强制工具调用：含数字的回答必须关联工具调用，否则拒绝输出

### Q9：如果从 0 设计 AI 客服 Agent？

**方法论：** 明确目标 → 场景拆解 → 技术选型 → 评测体系 → 迭代策略

| 步骤 | 内容 |
|------|------|
| 业务目标 | 替代多少人工？准确率底线？ |
| 场景拆解 | 高频简单 / 多轮复杂 / 需转人工 |
| 技术方案 | FAQ 用 RAG；操作类用 Tool Calling；复杂类多步规划 + 人工兜底 |
| 评测体系 | 准确率、解决率、满意度、转人工率 |
| 迭代策略 | 先上 FAQ，再上复杂场景 |

**以长桥为例：**

| 场景类型 | 方案 |
|---------|------|
| FAQ（怎么入金、交易时间） | RAG 检索帮助中心 |
| 操作类（查持仓、撤单） | Tool Calling 调内部 API |
| 合规敏感类（账户冻结、投资建议） | **一律转人工** |

> 和普通客服最大区别是 **合规红线** — 费率说错可能被投诉，推荐个股更是违规。准确率目标 95%+。

### Q14：RAG 工作流程？

```
离线阶段：文档 → 切分 → Embedding 向量化 → 存入向量数据库
在线阶段：用户查询 → 向量相似度检索 → 取 Top-K 片段
生成阶段：检索结果 + 用户问题 → 拼入 Prompt → LLM 生成回答
```

**关键设计点：** chunk 大小、Embedding 模型、Top-K 数量、Prompt 模板

> 我的核心问题是跨语言匹配（中文查询 vs 英文年报），通过切换中英双语 Embedding 解决。

---

### Q15：为什么选 Top-2？

- 分索引已做粗筛，单股 300-500 chunk，Top-2 命中率已很高
- 实测 Top-2 和 Top-3 差异不大
- 具体 K 值根据场景调：单一事实用 K=2，多角度分析可加到 K=5

---

### Q16：为什么 chunk_size=800, overlap=100？

- **chunk_size=800：** 10-K 段落长，一个风险因素描述 500-1000 字符，800 让大多数段落完整
- **overlap=100：** 约 1-2 句话，防止边界信息切断
- **separators 段落优先：** 按 段落→换行→句子 逐级降级，尽可能在自然边界切分
- 这组参数是实测后"在 10-K 场景下效果最好"的经验值

---

### Q17：Prompt 设计思路？

Prompt 模板只有 **6 条精简规则**：角色定义、多轮记忆、来源引用、禁止编造、数字格式化、RAG 路由。

> 核心原则：**"少即是多"** — 规则越多 LLM 越容易忘，能用机制保障的不依赖 Prompt。

---

### Q18：Tool Calling 和 Function Calling 区别？

本质相同。都是让 LLM 决定调用哪个外部工具并生成参数。LLM 不直接执行工具，而是输出"我想调用 X 工具，参数是 Y"的结构化指令，由外部编排层执行后将结果返回 LLM。

---

### Q19：Embedding 模型怎么选？

综合考虑：**语言**（纯中/纯英/跨语言）× **维度**（越大越准但越贵）× **速度**（在线/离线）× **领域**（通用/垂直）

## 三、产品能力

---

### 做过最失败的需求

**S：** 信息披露平台有一个"档案固化"需求 — 监管要求对每日披露内容固化存档，保存两年。

**T（我的错误做法）：** 机械地把"每日固化"理解为"每天全量保存一份快照"，没区分数据类型。

**A（问题暴露）：** 上线初期没问题，但业务增长后磁盘占用急剧膨胀。

**根因：** 没深入分析数据特征——实际上披露数据分两类：
- **注册类**（机构信息、资质）：变更频率极低，可能几个月不变
- **交易类**（每日交易结果、公告）：每天不同，必须每日固化

全量固化 = 大量注册类数据每天重复保存 = **巨大存储浪费**

**解决方案：** "增量固化、全量查询" — 只对每日有变更的数据增量固化，查询时通过时间戳拼接还原。

**R：** 存储占用下降 **~70%**，完全满足监管要求。

> **启示：**
> 1. 需求理解不能停留在字面意思
> 2. 产品设计要理解数据特征
> 3. 方案评审要拉上技术

---

### 做过最成功的需求

**S：** 信息披露平台需支持多种披露类型，且每隔几个月就有新合规要求。

**T：** 每次新增都开发写代码，排期 1-2 周，跟不上合规节奏。

**A：** 抽象出底层 **配置化架构** — 模板配置、审核流程配置、短信通知配置、披露项字段配置。管理员后台配置即可新增类型，不需要改代码。

**R：** 后续新增全部通过配置实现，**开发零介入，上线时间从 1-2 周 → 1-2 天**。

---

### 怎么做优先级排序？

1. 按 **合规时间节点** 分批（必须上线 vs 可后续迭代）
2. 按 **用户使用频率**（高频查询 > 低频管理）
3. 按 **技术依赖关系** 排序

---

### 对产品的认识

- 持续关注产品的数据指标
- 产品设计要识别真实需求，最终贴合常识
- 做好标准化和个性化的平衡，迭代兼顾完整性与扩展性

---

### 从 0 到 1 踩过最大的坑

**坑：** 低估跨系统数据一致性。平台需汇集 7 个业务系统数据，我没逐一确认数据定义和更新频率。上线后发现各系统数据口径不一致，同一指标定义不同，前台数据互相矛盾。

**根因：** 默认同名字段含义一致，没做数据源摸底。

**改进：** 此后每个项目启动阶段做 **数据源摸底表**（字段映射、口径定义、更新机制、异常处理）作为需求文档必填项。后续数据准确性问题降为零。

> ⚡ **从 0 到 1 最大的风险不在开发阶段，而在需求阶段。**

处理顺序：**止损 → 担责 → 复盘 → 改进**

## 八、AI/LLM/RAG/Agent 技术八股

---

### Agent 主流架构

| 架构 | 特点 |
|------|------|
| **ReAct** | Thought→Action→Observation 循环，可解释性强但格式遵循要求高 |
| **Tool Calling** | 模型原生支持，稳定性高但依赖模型能力 |
| **Plan-and-Execute** | 先规划再执行，适合复杂多步任务 |
| **Multi-Agent** | 多 Agent 协作（AutoGen、CrewAI） |

> 我的实测：能用 Tool Calling 就不用 ReAct。

---

### RAG 常见失败模式

| 失败模式 | 说明 |
|---------|------|
| 检索失败 | 查询和文档语义不匹配（我遇到的中英跨语言问题） |
| 生成幻觉 | 检索正确但 LLM 自由发挥 |
| 上下文溢出 | 检索太多 Chunks 超出 Context Window |
| 知识过期 | 向量库未及时更新 |
| 切分错误 | chunk 边界切在关键信息中间 |

---

### Prompt Engineering 核心技巧

1. **角色设定：** 给 LLM 明确角色和行为边界
2. **Few-shot 示例：** 提供正确输出示例
3. **结构化输出：** 要求 JSON/Markdown 固定格式
4. **Chain-of-Thought：** 要求分步推理
5. **约束规则：** 明确不能做什么

> 最大教训：**Prompt 不是万能的。** 效果不好时要思考问题是否该在其他层面解决。

---

### Token 是什么？怎么影响产品设计？

Token ≈ 0.7 个中文字。影响：
1. Context window 限制了单次输入量（需截断/摘要）
2. 输出 token 限制了回答长度
3. 成本按 token 计费（需控制调用频率）

---

### 数据飞轮

用户使用 → 产生日志 → 标注好/坏 case → 优化检索/Prompt → 效果提升 → 更多用户使用

> 关键：设计低成本反馈收集机制（如"有帮助"按钮）

---

### 如何控制 LLM 输出质量？

**五层防御（从轻到重）：**
1. Prompt 约束
2. 输出格式校验（JSON Schema）
3. 后处理过滤（敏感词/格式）
4. 数据层强制（来源标注）
5. 人工审核

---

### AI PM vs 传统 PM

传统 PM：定义确定性需求 + 拆解功能 + 追交付

AI PM **额外需要**：
- 管理不确定性（效果无法保证）
- 设计评测体系
- 理解技术边界做可行性判断
- 设计人机协作兜底方案

**AI PM 关键能力：**
1. 评测先行 — 自己设计评测方案
2. 懂技术边界 — 对 LLM 能力有自己的理解
3. 能上手验证 — 会用 AI Coding 验证创意
4. 跨团队推动 — 协调研发、设计、算法、合规

---

### RAG 优化路径（按收益排序）

1. **Embedding 模型选型**（我的实测：效果差 40%）
2. **Chunk 策略优化**（语义切分 > 固定长度）
3. **Query Expansion / HyDE**（用 LLM 改写查询）
4. **Reranker 精排**（Cross-encoder 二次排序）
5. **Hybrid Search**（向量 + BM25 融合）

---

### Agent 评测维度

1. 工具路由准确率
2. 参数生成准确率
3. 多步规划成功率
4. 幻觉率
5. 延迟与稳定性

---

### LangChain vs LlamaIndex

- **LangChain：** 通用 Agent 框架，组件丰富，适合编排复杂工作流
- **LlamaIndex：** 专注 RAG，数据索引/检索体验更好
- **选择：** 纯 RAG 用 LlamaIndex，需要 Agent + 工具调用用 LangChain

---

### 开源 vs 闭源

| 维度 | 开源（DeepSeek/Llama） | 闭源（GPT-4/Claude） |
|------|----------------------|---------------------|
| 成本 | 推理便宜，部署需 GPU | 按 token 付费 |
| 隐私 | 数据不出境 | 依赖厂商政策 |
| 定制 | 可微调 | 只能 Prompt/RAG |
| 效果 | 接近（某些场景超越） | 综合最强 |

---

### Prompt vs Fine-tuning vs RAG

| 方案 | 特点 |
|------|------|
| **Prompt** | 最轻量，调整行为/格式，无法注入新知识 |
| **Fine-tuning** | 中等投入，风格统一/领域适配，需要标注数据 |
| **RAG** | 知识增强/实时更新/来源追溯，无需训练 |

---

### 成本模型

```
单次调用成本 = (输入 tokens × 输入单价) + (输出 tokens × 输出单价)

DeepSeek Chat 示例：
- 输入: ¥0.001/1K tokens
- 输出: ¥0.002/1K tokens
- 一次 RAG 查询（2000 in + 500 out）≈ ¥0.003
- 日调用 10000 次 ≈ ¥30/天 ≈ ¥900/月
```

**成本优化策略：**
1. 缓存高频相似问题（省 60-80%）
2. 简单问题小模型、复杂问题大模型（Router）
3. 控制 context 长度
4. 异步批处理非实时需求
