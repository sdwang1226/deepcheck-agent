## 二、核心技术决策

### 2.1 为什么选 Tool Calling 而非 ReAct

| 维度 | ReAct | Tool Calling（选用的）|
|------|-------|----------------------|
| 调用方式 | LLM 写文本 → 正则解析 | LLM 输出 JSON → 直接调函数 |
| 并行调用 | ❌ 不支持 | ✅ Round 1 并行调了行情+财报 |
| DeepSeek 适配 | ❌ 经常格式错误 | ✅ 稳定 |
| 决定理由 | — | DeepSeek 对 ReAct 格式遵循差，Tool Calling 更可靠 |

### 2.2 为什么用 Prompt 约束而非 Fine-tune

```
选 Prompt Engineering 的理由：
1. DeepSeek 不支持 fine-tune
2. MVP 阶段快速迭代，改 Prompt 比重训模型快
3. 评测结果证明 Prompt 约束 + 工具模板已能达到 90%+ 的来源引用率
```

### 2.3 RAG 方案选型

| 方案 | chunking | embedding | 维度 | 命中率 | 结论 |
|------|----------|-----------|------|--------|------|
| v1: 机械切分 + TF-IDF | 512 char | TF-IDF | — | 低 | 只适合英文精确匹配 |
| v2: 章节切分 | 按 Item 标题 | TF-IDF | — | 低 | 正则切分不可靠，放弃 |
| v3: 标准切分 + MiniLM | Recursive(800) | all-MiniLM-L6-v2 | 384 | 50% | MVP 初版 |
| **v4: BGE-large-zh** | Recursive(800) | **BAAI/bge-large-zh-v1.5** | **1024** | **70%** | **当前版本** ✅ |
| v5(规划): 混合检索 | 同上 | BGE + BM25 rerank | 1024 | 预计 85%+ | V1.0 目标 |

**v3 → v4 升级说明：**
- 向量库：ChromaDB → FAISS（ChromaDB 在 Windows 上 Rust binary 崩溃）
- 模型：MiniLM (384维/英文) → BGE-large-zh (1024维/中英双语)
- 索引架构：单股票 → 多股票按 symbol 分索引路由
- 数据源：Apple 1只股票 → 6只美股 TOP（AAPL/MSFT/GOOGL/NVDA/TSLA/AMZN）
- 命中率：基础测试 100%不变，高难度测试（中文查询+语义匹配）50% → 70%

**多股票索引架构：**
```
data/faiss_indices/
  ├── registry.json        # 索引注册表 (ticker → path + chunks)
  ├── AAPL_US/             # Apple FAISS 索引
  ├── MSFT_US/             # Microsoft FAISS 索引
  ├── GOOGL_US/            # Alphabet FAISS 索引
  ├── NVDA_US/             # NVIDIA FAISS 索引
  ├── TSLA_US/             # Tesla FAISS 索引
  └── AMZN_US/             # Amazon FAISS 索引
```

### 2.4 网络策略

```
DeepSeek API  → 走代理 (127.0.0.1:7897, 被墙)
Longbridge API → 直连 (NO_PROXY=open.longportapp.com, 香港服务器)
HuggingFace   → 走代理 (模型下载; 运行时 local_files_only=True)
SEC EDGAR     → 走代理 (10-K 年报下载)

实现: 全局 HTTP_PROXY/HTTPS_PROXY + NO_PROXY 白名单
```

### 2.5 环境兼容性注意事项

```
PyTorch:       必须 2.4.1+cpu (2.12版 c10.dll 加载失败)
transformers:  需要 monkey-patch check_torch_load_is_safe (torch 2.4.1 vs 要求 2.6+)
ChromaDB:      不可用 (Rust native binary crash)→ 已替换为 FAISS
onnxruntime:   1.19.2 (高版本报错)
```

## 三、数据流详解

### 3.1 单次对话流程

```
1. 用户输入（如"NVIDIA 有什么风险？"）
2. ConversationBufferMemory 提取历史对话
3. ChatPromptTemplate 组装完整 Prompt (system + history + user + scratchpad)
   - system prompt 中包含可用 10-K 股票列表
4. ChatDeepSeek 处理 Prompt → 决策调哪个工具 + 传什么参数
5. AgentExecutor 执行工具调用
   - 行情/财报工具: Longbridge API 直连
   - RAG工具: 根据 symbol 路由到对应 FAISS 索引 → 返回最相似的 2 个段落
6. 工具返回值 + 原始问题 → ChatDeepSeek 第二次调用 → 生成最终回答
7. 回答存入 ConversationBufferMemory
```
